文件与URL命名规范：分隔符使用指南

前言

在数字化工作中，无论是个人文件管理、团队项目协作，还是网站开发，采用一套统一、清晰的命名规范至关重要。良好的命名规范能极大提升信息的可读性、可维护性和可检索性。

本文档旨在提供一套关于分隔符（主要是下划线 _ 与连字符 -）使用的完整指南，并重点介绍一套适用于文件和文件夹命名的双层分隔符系统。

Part 1：通用分隔符泛用性对比 (_ vs. -)

下划线 (_) 和连字符 (-) 是最常用的两种分隔符，但它们的“泛用性”完全取决于应用场景。简单来说，连字符更适合面向公众和网络，而下划线更适合技术性、内部使用的场景。

各场景推荐与原因

场景 (Use Case) 推荐分隔符 主要原因
URL 网址 - (连字符) SEO 友好。Google 等搜索引擎将连字符视为单词分隔符（如 my-page 被理解为 my page），而下划线被视为单词连接符（my_page 被理解为 mypage），前者对搜索排名更有利。
文件名 (Filename) _ (下划线) 易用性与兼容性。在图形界面或命令行中，双击带下划线的文件名（file_name.txt）通常能选中整个名称，而带连字符的（file-name.txt）可能只选中一部分。
编程 (Programming) 取决于语言规范 - Python: 强制推荐下划线 snake_case (e.g., my_variable)。
- CSS: 普遍使用连字符 kebab-case (e.g., .my-class)。
- JavaScript/Java: 多用 camelCase，不直接使用分隔符。
数据库 (Database) _ (下划线) 可读性与安全性。snake_case 是绝大多数数据库设计的标准实践。连字符在某些 SQL 环境中可能被视为非法字符或需要特殊处理，容易出错。

Part 2：最佳实践：双层分隔符文件命名系统

对于需要结构化管理的文件和文件夹，我们推荐使用一套分级、双层的分隔符系统。这套系统能最大化提升文件命名的清晰度和机器可读性。

核心原则

  1. 主分隔符 (Major Separator): _ (下划线)

    • 用途：用于分隔文件名中不同的功能区块（元数据字段）。它像一个“宏观”分隔符，用来划分文件名的主要组成部分。
    • 示例：在 日期、项目名、文件描述 等核心区块之间使用。

  2. 次分隔符 (Minor Separator): - (连字符)

    • 用途：用于连接一个功能区块内部的多个单词，使其形成一个逻辑上统一的短语。
    • 示例：项目名称为 “Project Phoenix”，在文件名中应表示为 project-phoenix。

标准命名模板与示例

一个健壮且通用的文件名模板如下：

模板: [日期]_[项目或类别]_[文件描述]_[版本号].[后缀]

实际应用示例:

20250606_project-phoenix_marketing-launch-plan_v1.2.docx

文件名结构解析:

  • 20250606: 日期区块。使用 YYYYMMDD 格式确保文件能按时间正确排序。
  • _: 主分隔符。分隔「日期」与「项目名」。
  • project-phoenix: 项目区块。内部使用 - 将单词连接成一个完整的项目名。
  • _: 主分隔符。分隔「项目名」与「文件描述」。
  • marketing-launch-plan: 文件描述区块。内部使用 - 连接多个单词，形成清晰的描述短语。
  • _: 主分隔符。分隔「文件描述」与「版本号」。
  • v1.2: 版本号区块。v 是 “version” 的缩写。
  • .docx: 文件扩展名。

双层分隔系统的优势

  • 结构清晰，可读性高：任何人都能通过分隔符快速理解文件名的结构和含义。_ 划分大类，- 连接细节。
  • 排序与分组能力强：在文件管理器中，文件可以自然地按日期、项目名等区块进行排序和逻辑分组。
  • 便于脚本自动化处理：结构化的命名使得编写脚本（如 Python, Shell）进行批量重命名、归档、数据提取等操作变得异常简单。
  • 跨平台兼容性：此命名法在 Windows, macOS, Linux 等所有主流操作系统中都表现完美，不会产生兼容性问题。

总结

  • 在需要单一分隔符的场景下，根据具体用途（Web 或本地）选择 - 或 _。
  • 在需要结构化命名的文件管理场景中，强烈推荐采用 _ 作为主分隔符、- 作为次分隔符的双层系统，它将为你的数字资产管理带来长期的便利和清晰度。
